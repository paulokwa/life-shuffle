## 2026-06-12 — Flutter app built end-to-end: local planner + Firebase auth

- **Goal**: Build the full local planner loop and add Firebase/Google sign-in as an auth gate.
- **Summary**: Four milestones were completed in this session. The Flutter UI shell was built first (theme tokens, 8 shared widgets, 5-tab nav, Today/Plan/Activities/Progress/Settings/Onboarding screens). Then the rule-based weekly planner was wired in (Activity/DayPlan models, PlannerService, AppState via InheritedNotifier). Then SharedPreferences persistence was added so enabled states, check-ins, locks, and the regen seed all survive page refresh. Finally Firebase core + Firebase Auth were added with Google sign-in and an AuthGate that falls back to local-only mode when Firebase isn't configured.
- **Files changed** (key ones):
  - `lib/main.dart` — async init, Firebase setup, AuthGate routing
  - `lib/firebase_options.dart` — real credentials for project `life-shuffle-8d3bd` (generated by flutterfire configure)
  - `lib/widgets/auth_gate.dart` — signed-out → SignInScreen, signed-in → onboarding → BottomNavShell
  - `lib/screens/sign_in_screen.dart` — Google sign-in UI
  - `lib/services/auth_service.dart` — Firebase Auth wrapper
  - `lib/services/planner_service.dart` — static weekly plan generator with seeded Random
  - `lib/services/persistence_service.dart` — SharedPreferences read/write
  - `lib/state/app_state.dart` — ChangeNotifier + InheritedNotifier scope for all mutable state
  - `lib/screens/` — Today, Plan, Activities, Progress, Settings, Onboarding (all live)
  - `pubspec.yaml` — google_fonts, shared_preferences, firebase_core, firebase_auth
- **Decisions made**:
  - No Riverpod/Provider — pure Flutter InheritedNotifier.
  - Planner data stays local (SharedPreferences). Firestore sync is a future milestone.
  - Firebase auth is web-first (signInWithPopup). Local-only mode still works if Firebase config is missing.
  - `AuthService.isReady` static flag gates all Firebase calls so the app degrades gracefully.
- **Tests run**: `flutter build web` passes clean. `flutter analyze` — 0 errors, 0 warnings, 48 pre-existing infos (prefer_const, withOpacity). Google sign-in manually verified working in Chrome.
- **Current state**: The app is fully runnable. Sign in with Google → onboarding → planner loop (Today, Plan, Activities, Progress). Check-ins persist across refresh. Plan regeneration with seed works. Lock/unlock works. Sign-out returns to sign-in screen.
- **Next recommended step**: Firestore sync — write weekPlan and check-in state to Firestore so data persists across devices and sets up shared editing for Kwame and Laura.
- **Open questions**:
  - Should Firestore be the primary store (replacing SharedPreferences) or a secondary sync layer?
  - Should both Kwame and Laura share one Firestore document or have separate user documents merged at display time?

## 2026-06-13 — Firestore sync foundation

- **Goal**: Add minimal Firestore sync foundation while preserving local fallback.
- **Summary**: Added `cloud_firestore` and implemented `FirestoreSyncService`. Integrated sync into `AppState` so that when a user is signed in, their state (seed, enabled activities, check-ins, locked states) is persisted to Firestore. SharedPreferences remains as a local fallback and cache. `AuthGate` was updated to notify `AppState` of user auth changes.
- **Files changed**:
  - `pubspec.yaml` (added `cloud_firestore`)
  - `lib/services/persistence_service.dart` (added serialization to `SavedState`)
  - `lib/services/firestore_sync_service.dart` (new service for Firestore operations)
  - `lib/state/app_state.dart` (added sync logic and user ID management)
  - `lib/widgets/auth_gate.dart` (added auth listener to trigger sync)
- **Decisions made**:
  - Use a single document per user at `users/{userId}/calendars/default` for MVP.
  - `AppState` is the coordinator for both local and remote persistence.
  - Sync is triggered on sign-in; local state is overwritten by Firestore state if available.
- **Tests run**: `flutter test`, `flutter analyze`, `flutter build web`.
- **Current state**: Firestore sync foundation is in place. App correctly handles signed-in vs local-only modes.
- **Next recommended step**: Shared editing — allow Kwame and Laura to point to the same calendar document or implement a sharing mechanism.
- **Open questions**:
